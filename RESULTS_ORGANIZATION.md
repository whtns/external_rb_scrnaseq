# Results Directory Organization

This document maps how output files are organized in the `results/` directory.

## Current Organized Subfolders

These subfolders are actively used and should be preserved:

| Folder | Contents | Notes |
|--------|----------|-------|
| `filtering_timeline/` | `{sample}_filtering_timeline_{extension}.pdf` | Sample filtering QC plots |
| `subset_karyogram/` | `{sample}_subset_karyogram.pdf` | Karyogram plots for filtered samples |
| `celltype_plots/` | `{sample}_dimplot.pdf`, `{sample}_featureplot.pdf` | Cell type visualization plots |
| `clustify/` | `{sample}_clustifyr.{pdf,csv}` | Automatic cell type assignment results |
| `speckle/` | `{group}.{pdf,xlsx}` | Cell type composition analysis |
| `effect_of_filtering/` | Hierarchical (abbreviation/, distribution/) | Pre/post-filtering comparisons |
| `*_clustree_diffex/` | Clustree differential expression | all/, cis/, trans/ variants |
| `*_enrichment_plots/` | Enrichment visualization | cis_enrichment_plots/, trans_enrichment_plots/ |

## Files in Root (results/)

These are generated by various functions. When refactoring, organize into:

## Functions Writing to results/

Key functions that write directly to results/ (from numbatHelpers):

**figure_functions.R:**
- `plot_fig_*()` → Main figures
- Various supplemental figure functions

**plot_functions_*.R:**
- Per-sample plots and analyses
- Heatmaps, UMAPs, markers, QC plots

**enrichment_functions_*.R:**
- Enrichment tables and plots

**diffex_functions_*.R:**
- Differential expression results

**numbat_functions_*.R:**
- Clone analysis and summaries

**integration_functions.R:**
- Integration analysis outputs

## Guidelines for New Code

When adding new file writes to `results/`:

1. **Check existing organization** - Use existing subfolders if they match your file type
2. **Group by analysis type** - Don't scatter related files across root
3. **Use descriptive names** - Include sample ID, analysis type, and parameters
4. **Document the pattern** - Update this file when establishing new patterns

## Migration Path

Gradual migration is preferred:
- No breaking changes to existing workflows
- Update functions as they're refactored for other reasons
- Document new subfolders as they're created
- Archive old root files when subfolders are established
